\section{Requirements and design goals}

The literature reviewed in chapter 1 identifies a gap in the comparative empirical evaluation of management polices for ASP-mediated UAS Traffic management (UTM), closing that gap requires a simualtion framework that is sufficialtly faithful to the operational context of UTM to make its results meaningful, and sufficiently flexible to support controlled comparison across a range of policies and operating conditons. This section sets out the requirements created with these considerations in mind. Which together infrom the design choices documented in the remainder of the chapter. \par

\vspace{1em}

The functional requirements are as follows. The framework must model an urban airspace as a network of independent Airspace Service providers (ASPs), each controlling thier defined area and handling the handover of traffic to its neighbouring controllers. It must process flight plan requests dynamically during execution rather then as a fixed batch known in advance (to reflect the real world) it must support multiple management from MAPF reviewed in the chapter above. it must permit policies to be tested in the same envrionments and configurations. it must record sufficent data on each flight and simulation to support the saftey, efficency and coordination metrics described. \par

\vspace{1em}

The non functional requirements are equally importatnt. The framework must be deteminsitic given a fixed random seed, so that experimental results are reporducible. It must be modular enough to permit new polices to be added without modifying the siulation core, in support of both the comparitive evaluations undertaken in this work and any future extensions. It must provide configeration through external files rather then paremeters so experimental conditions can be varied without recompliation. It must complete experiments within a resanable time on comodity hardware as it will be ran many time in many configeration. \par

\vspace{1em}

From these requirements they derive several other design goals that reoccur throughout the rest of this chapter. The simulation must be discrete event driven rather than time stepped. Many considerations motivated this choice. First, performance: a time-stepped simulation processes every agent (UAV) at every clock tick regardless of whether there state has changed, while a discrete-event simulation processes a UAV only when an event affects it, which scales considerably better as the number of UAVs and the length of the simulated horizon grow. Secondly, precision in collision detection: continuous time geometric collision detection over analytic trajectory segments yields the exact moment and distance of closest approach, while a fixed-tick simulation can miss close-approach events that occur entirely within a single time step unless the step is made impractically small. Third, reproducibility: a discrete-event simulation is fully determined by its random seed and configuration, whereas a time-stepped simulation introduces the time step itself as a hidden parameter whose value can affect experimental results. SimPy \cite{simpy2024} is the natural choice for this paradigm in Python. \par

\vspace{1em}

These requirements and design goals together motivate the architecture presented in Section 2.2, the simulation model described in Section 2.3, the policy abstraction documented in Section 2.4, the reservation mechanism developed in Section 2.5, and the project management approach summarised in Section 2.6.

At the top of the archiecture is a single simulation object. It serves as the orchestrator and it is constructed from the configuation file. It is responsable for instatiating the SimPy envrionment, the metrics collector, creating the plannng polcies and the world manager. Once instantiated it delegates the running of the simulation event loop to a SimPy event manager after this its only responsable for recording the final metrics to a file on completion.

The world manger owns the world strucuter and manger the inter-airspace coordinatino. It loads a world manifest that specifies the layout of airspacesand the connectivity between them. From the given manifest it also initiates an Airspace object per sector. Once the airspaces have been created they are represented as nodes in a graph and the gates are added as edges. This is what is used when a UAV requres a route that crosses airspace boundires and it returns a sequesnce of airspaces and gates that the local planner within each airspace uses to get the UAV from its current position to a gate. 

Each Airspace repersents one sector of the simulated airspace. It owns per-depot job queues populated by job jeneration processes (described in Section 2.3). A set of Simpy resource objects representing depots and gates as capcity constrained resources. The fleet of UAVs whos depot is within the sector. Each airsapce also instansiates an ASP (air service providor) object that manages the sectors dynamic state from tracking uav poistion to continous collision detection through segment reservation from trajectories (described in Section 2.5). The static spatial strcuture of the envrionment (obstacles, geomatry and depots) is held in the GridMAp together comprise the environment layer in the architecture. Both the static and dynamic environment help impliment the flight plan mangement role that UTM literature attributes to a serice provider \cite{kopardekar2014utm, faa2022utmconops}.

UAV agents (which belong to airspaces), each run a SimPy process with its own state machine. A UAV holds a reference to its current mangement policy, its home depots job queue and its parent airspace along with a set of configurations loaded from a configuration file including but not limited to maxiumum speed, saftey radius and acceleration. When idle, a UAV yeilds until its home depot has a free job it can pick up. When a uav picks up a job this is when the state machine kicks into affect. It requests a route through global airspace which returns a sequence of waypoitns including the start point, gates to other airspaces and the destination. Then the UAV will register with the ASP (Air Space Providor) to query for predicted collisions and update its own trajectory over time. The full lifcyle of the UAV and its eight state machine will described in Section 2.3.

The MAPF planner is an abstraction that every policy fits behind. Every policy must inherit the common interface which allows for easy plug and play in the configuation. This means that Greedy, BFS, DFS, A*, Occupancy A*, Prioritsed A*, Cooperative A* and WHCA* all inherit velociy calculators and can return a list of waypoints. The Factory classes is able to be given a poilcy identifier from the configeration and instantiates the corresponding policy. This decoupling is essential for the comparative evaluation reported in Chapter 4, since it allows polices to be exchnaged without modification to the airspace, UAV, or metrics subsystems. 

The metrics subsystem records per-flight, per-collision, and per-run data during simulation and writes the results to CSV. Configuration is loaded from external Json files and can be modified by sending a config.py data object or a command line arugument when running main. Exernalising the configeration this way allows for for testing without chaninging the source code such as when giving the simulaiton a variaty of seeds, run times, polices and other configerations evaluated in Chapter 4. 

The chioce to model the simulation as a discrete-event-driven system, rather than as a time-stepped one, is fundamental to the architecture and is justified in Section 2.1. At the architectural level, this chioice is reflecetd in the use of SimPy as the simulatino core. This is because simpy processes are routines that can yeild control to the event loop at decision points, producing a simualtion in which simulated time advances directly between events rather than at fixed clock intervals. This in turn allows continous time collision detection within the ASP, removig the time quantisation issues as dt would impact collision detection. 

\section{simulation model}

This section describes the dynamic behavoir of the simulation; how UAVs behave over time, how jobs are generated, how routes are produced and how conflicts are detected and resolved. This will explain how the static archiecture in 2.2 provides a strucutre in which all of the simulation takes place. 

\subsection{descrete event excection}

The simulation uses SimPy at its foundations, treating each UAV, gate and job generator as a yeild control or decision point in an event loop. This means simulated time advances directly between events, therefor when there is no UAV making a decision the event loop jumps forwards to the next scheduled event such as a UAV reserving a new segment once commpleting its last jump or a new job getting generated. This is the principles behind discrete event driven simualition.

There are two key implications of this approach. Firstly there is no fixed simulation tick, meaning no DT parameter that controls the granularity of state updates and adding one would not change behaviour becuase the simulator processes events at the exact time they occour. Secondly, two events at the same point in time are ordered deterministically by SimPys internal event-priority mechanism. This means that given the same seed if you ran the same simulation configeration it will give the exact same solution (This determism was also varified during experimental procedure in Chapter 4).
